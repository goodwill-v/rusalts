# Model routing (main → heavy): как работает и как настраивать

## Зачем это нужно

Policy “main → heavy” позволяет держать **стоимость и задержку** под контролем:
- большую часть времени работает более дешёвая/быстрая **main** модель;
- дорогая **heavy** включается только когда риск ошибки выше (регуляторика/комплаенс/неуверенность) или результат не прошёл валидацию.

## Где включается и чем управляется

### Переменные окружения

Задаются в `.env` (см. `.env.example` и `deploy/env.production.example`):

- **Backend**
  - `BACKEND_MODEL_MAIN`
  - `BACKEND_MODEL_HEAVY`
- **Content**
  - `CONTENT_MODEL_MAIN`
  - `CONTENT_MODEL_HEAVY`

Логика выбора (policy) находится в `app/model_routing.py`.

## Триггеры эскалации (main → heavy)

### Backend (чат / синтез по KB)

Эскалация на heavy происходит если:
- **Legal/комплаенс‑триггер**: в тексте запроса встречаются ключи (см. список ниже).
- **Низкая уверенность по KB**: при синтезе по найденным статьям ровно **1** KB‑хит (слабый RAG) — повышаем риск ошибки, поэтому включается heavy (если задан `BACKEND_MODEL_HEAVY`). При **0** хитах сначала идут whitelist‑источники и веб‑fallback: там по умолчанию используется **main**, а **heavy** — только на ретрае после пустого ответа модели.
- **Валидация результата**: модель вернула пустой ответ → один ретрай на heavy.

**Список ключей (backend legal trigger)** — правится в `app/model_routing.py`:
- `152-ФЗ` (и варианты написания)
- персональные данные / `ПДн`
- Роскомнадзор / `РКН`
- согласие на обработку
- локализация данных
- трансграничная передача
- оператор персональных данных
- реестр / уведомление
- комплаенс
- закон / постановление / приказ / регламент

### Content (публикации)

Эскалация на heavy происходит если:
- change package содержит элементы с `classification` начинающимся на `legal.` (например `legal.regulatory`)
- **Валидация формата**: ожидается JSON‑структура (title/site_text/vk_text/internal_note), но ответ не удалось распарсить → один ретрай на heavy.

## Что логируется (для контроля затрат)

В JSON‑логах сохраняются:
- `model`
- `tokens_in`, `tokens_out`
- `purpose`
- `routing_reason` (почему выбрали main/heavy)

Это позволяет прозрачно видеть, **когда и почему** включалась дорогая модель.
