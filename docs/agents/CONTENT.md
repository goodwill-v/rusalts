# Агент 4: Content (релизы, посты, публикации)

## Цели

- превращать изменения из `change_package.json` в понятные релизы/посты;
- публиковать на сайт АЛЬТЕРНАТИВА и в VK (со стилевой адаптацией);
- фиксировать “что ушло в прод” по продукту и БЗ.

## Входы

- `data/changes/YYYY-MM-DD/change_package.json` от Parser‑агента
- релизы продукта (backend/frontend) — список изменений от команды
- тональность/гайдлайны бренда (будет расширяться)

## Выходы

- новость для сайта (HTML/Markdown → публикация)
- пост для VK (коротко, с CTA)
- внутренняя заметка “для команды” (более технически)

## Форматы публикаций (MVP)

### Порядок публикаций 
Обработанные материалы для сайта и публикации Вконтакте первоначально отправляются на почту шефу Chief v.devops@yandex.ru. Chief обратным письмом отправляет решение по публикации:
- «ДА: «Текст публикации»- одобряет публикацию. Тексты отправляются в ленту Вконтакте, на сайт в соответствующий раздел.
- «НЕТ: «Текст публикации» - соответствующий текст сохраняется в архиве на 30 дней без публикации. После удалаяется
- «РЕДАКТИРОВАТЬ, пояснение: «Текст публикации»  - соответствующий текст возвращается для редактирования в соответствии с пояснением.

### Общий формат публикаций
Каждой новости и публикации присваивать идентификационный номер, надо публиковать его рядом с новостью после даты в скобках, 5 знаков: "05.04.2026 (00035)"
Формат публикации:
Название, дата (номер)
Анонс (до 200 знаков) виден в ленте ВКонтакте. Это начало текста, которое вызывает интерес у читателя и отображает суть материала.
Текст  новости. Открывается при нажатии «Показать ещё» 
Ссылка на источник.

### Сайт
Структура:
- заголовок
- 3–7 буллетов “что изменилось”
- блок “важно для организаций/разработчиков/пользователей”
- ссылки на официальные источники
- дата и таймзона (UTC + MSK при необходимости)

### VK
Правило: коротко, 1–2 экрана, затем ссылка на сайт/виджет.

## Публикация в VK: что понадобится в `.env`

Нужно проверить наличие/добавить (если будем публиковать через API VK):
- `VK_GROUP_ID` (id сообщества)
- `VK_WALL_ACCESS_TOKEN` или путь получения токена (права: `wall`, `docs`, `photos` — зависит от формата)
- `VK_API_VERSION` (например `5.199`)

Сейчас в `.env` есть `VK_SERVICE_TOKEN`, но для постинга на стену обычно используется токен сообщества/пользователя с нужными правами (модель авторизации уточним при реализации).

## Definition of Done

- каждый релиз содержит ссылки на источники
- чувствительные темы (законы/правила) — с пометкой “информационно” и без юридических советов
- публикации логируются (id поста/URL, timestamp, request_id)

## Краткий бриф (для раздачи роли “Content”)

Ежедневно делать релизы/посты по:
- `data/changes/YYYY-MM-DD/change_package.json` (изменения от Parser)
- изменениям продукта, ушедшим в прод (из команды)

Формат: готовить 3 версии:
- **Сайт**: подробнее (что изменилось / кому важно / ссылки)
- **VK**: коротко + CTA + ссылка на сайт/виджет
- **Внутренняя заметка**: технично (для команды)

Правила:
- legal‑темы: “информационно”, без юридических советов, со ссылками на официальные источники
- фиксировать дату и источники
