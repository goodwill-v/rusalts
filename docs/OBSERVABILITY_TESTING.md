# Логирование, мониторинг, тестирование (в т.ч. нагрузочное)

## Логирование (единый стандарт)

Цели:
- отладка интеграций (VK, RouterAI, парсер),
- аудит изменений БЗ,
- учёт стоимости токенов,
- инциденты и воспроизводимость.

Рекомендуемый формат: **JSON‑логи** (одна строка = одно событие).

Обязательные поля:
- `ts` (UTC, ISO‑8601)
- `level`
- `service` (`backend|parser|content|frontend`)
- `event`
- `request_id` (корреляция)
- `user_id` (если есть, иначе `null`)
- `kb_article_id` / `source_id` (если относится)
- `model` / `tokens_in` / `tokens_out` / `cost_usd` (для LLM)

## Тестирование

### Unit/Integration
- `pytest` для Python‑логики (парсер, классификация, формат change package).
- `fastapi.TestClient` для API‑контрактов.

Минимальный набор тестов:
- `GET /health` возвращает `{"status":"ok"}`
- `POST /api/chat` валидирует вход и возвращает ответ
- `GET /api/document-templates` возвращает массив `items`

### Контрактные тесты
- формат `change_package.json` (JSON Schema)
- формат KB‑статей (валидатор YAML‑шапки)
- формат триггеров/шаблонов (валидатор JSON)

## Нагрузочное тестирование

Цели:
- проверить стабильность `/api/chat` под нагрузкой,
- проверить время ответа (P95/P99),
- контролировать стоимость (LLM вызовы).

Рекомендуемые инструменты:
- **k6** (JS, удобно в CI)
- или **Locust** (Python), если проще команде

Сценарии:
- “частые вопросы”: короткий текст (1–2 предложения)
- “сложный вопрос”: длинный текст (2–4k символов)
- “всплеск”: 50–200 RPS на 2–5 минут

Метрики:
- latency P50/P95/P99
- error rate
- средние токены на запрос
- cost/request
