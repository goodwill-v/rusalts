# Карта проекта АЛТ‑эксперт

## Цель продукта

АЛТ‑эксперт консультирует пользователей по вопросам использования российских мессенджеров (приоритет — **MAX**):
- миграция с зарубежных платформ (стратегии, чек‑листы);
- шаблоны действий и документов;
- актуальные изменения требований/законов/правил;
- ежедневные релизы новостей и изменений.

## Ключевая идея архитектуры

Единый продукт собирается из 4 автономных сервисов/агентов + роль тимлида:
- **Backend**: API приложения, поиск по БЗ, интеграция с RouterAI, интеграции каналов (VK виджет, сайт).
- **Frontend**: UI виджета/страниц сайта АЛЬТЕРНАТИВА + коммуникационные каналы (VK mini app / виджет, далее — новые каналы).
- **Parser**: ежедневный мониторинг официальных источников (05:00 UTC), дифф, обновление БЗ, формирование пакета изменений.
- **Content**: релизы/посты по изменениям и релизам продукта, публикация на сайт и в VK.
- **TeamLead**: стандарты, схемы данных, регламенты, контроль качества.

## Предлагаемая структура репозитория

```
ALT/
  app/                         # FastAPI (виджет + API)
  data/                        # runtime данные (не коммитить), кроме sample-файлов
  knowledge_base/              # БЗ в структурированном формате + индексы
  templates/                   # шаблоны (JSON) + конфиги
  parser/                      # код ежедневного мониторинга (агент Парс)
  content/                     # генерация релизов и публикации (агент Конт)
  docs/                        # вся техдокументация по системе и агентам
  tests/                       # тесты API/логики
  loadtest/                    # нагрузочное тестирование (k6/locust)
  deploy/                      # nginx/systemd + синхронизация на сервер
  scripts/                     # утилиты (локальный запуск, разовые миграции)
```

## Сквозные потоки данных (end‑to‑end)

### 1) Вопрос пользователя → ответ

1. Пользователь задаёт вопрос (сайт / VK‑виджет).
2. Backend нормализует запрос, ищет в БЗ релевантные статьи/фрагменты.
3. Backend формирует “контекст” (выдержки + ссылки + дата актуализации).
4. RouterAI/LLM генерирует ответ по системному промпту + контексту.
5. Backend дополнительно:
   - предлагает шаблоны (по триггерам),
   - логирует обращение,
   - при необходимости создаёт тикет в поддержку (в будущем).

### 2) Ежедневный парсинг → обновление БЗ → релиз

1. Parser в 05:00 UTC собирает страницы/ленты из “официальных ресурсов”.
2. Делает дифф с предыдущим снимком, классифицирует изменения:
   - **законы/регуляторика** (приоритет),
   - **правила платформы/политики**,
   - **технические изменения (API, SDK, changelog)**,
   - **продуктовые новости**.
3. Обновляет БЗ по секциям и создаёт “пакет изменений” (machine‑readable).
4. Отправляет пакет агенту Content.
5. Content создаёт:
   - релиз на сайт,
   - пост в VK (адаптация стиля площадки),
   - внутренний changelog продукта (для команды).

## Правила качества (кратко)

- Источники: приоритет официальным ресурсам, все изменения в БЗ должны иметь ссылки на источник и дату.
- Юридические темы: только информационный характер, без “юридической консультации”.
- Экономия: модели LLM подбираются по “минимальной достаточности”; дорогие модели — только на сложные случаи.
