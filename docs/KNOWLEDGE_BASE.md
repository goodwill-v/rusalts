# База знаний: разметка, секции, обновление после парсинга

## Задача разметки

БЗ должна:
- позволять Backend‑агенту быстро находить ответы (по секциям, ключевым словам, датам);
- аккуратно отражать **изменения законов и правил** (хронология, источники, статус);
- быть обновляемой Parser‑агентом по секциям (без “перемешивания” информации);
- поддерживать “новости/релизы” для Content‑агента.

## Формат “атомарной статьи”

Храним как Markdown с YAML‑шапкой (легко diff’ить и парсить).
Каталог: `knowledge_base/articles/<section_path>/<id>.md`

Пример:

```text
---
id: max-migration-no-direct-import
section_path: MAX/Пользователям/Перенос_данных
title: "Почему нельзя перенести переписку «одной кнопкой»"
updated_at_utc: "2026-03-01T00:00:00Z"
keywords: ["миграция", "экспорт чата", "Telegram", "WhatsApp", "переписка"]
legal_relevance: "none"
sources:
  - title: "MAX: справка/FAQ"
    url: "https://max.ru"
---
Текст статьи...
```

## Специальный режим для регуляторики (важно)

Для секции `MAX/Регуляторика_и_комплаенс/*` вводим доп. поля:
- `legal_relevance`: `high | medium | low`
- `legal_status`: `draft | active | deprecated | unknown`
- `effective_from` (если есть)
- `jurisdiction` (например: РФ)

И требование:
- каждая статья должна иметь минимум 1 официальный источник;
- изменения фиксируются отдельным событием в журнале изменений.

## Журнал изменений (KB changelog)

Файл: `knowledge_base/changelog/YYYY-MM.md` (человекочитаемо)  
и `knowledge_base/changelog/YYYY-MM.jsonl` (машиночитаемо).

Запись JSONL:

```json
{"ts":"2026-04-06T05:12:00Z","type":"kb_update","article_id":"...","section_path":"...","change_package_item_id":"...","summary":"...","sources":["..."]}
```

## Как Parser обновляет БЗ (алгоритм)

1. Скачивает источники.
2. Делает нормализацию текста (удаление навигации/шумов).
3. Находит изменения (diff) относительно прошлого снапшота.
4. Классифицирует изменения в одну из категорий:
   - `legal.*` — регуляторика,
   - `tech.*` — API/SDK,
   - `product.*` — функциональность платформы,
   - `news.*` — объявления.
5. Для каждого изменения выбирает `kb_targets` (секция + статья) и:
   - обновляет существующую статью или создаёт новую;
   - пишет запись в `knowledge_base/changelog/*`;
   - добавляет item в `change_package.json` для Content‑агента.

## После первого парсинга

Проверка (“системная инспекция”):
- валидность метаданных статей (id/section_path/updated_at_utc/sources);
- отсутствие дублей (одна и та же тема в нескольких местах);
- наличие отдельной хронологии для `legal.*`;
- покрытие “шаблонов” (что просит пользователь чаще всего).
