# Что делаем после первого парсинга (инспекция системы)

## Цель

После первого ежедневного прогона Parser (05:00 UTC) провести системную инспекцию:
- качество классификации изменений,
- корректность попадания в секции БЗ,
- пригодность change package для публикаций,
- стоимость и производительность.

## Чек‑лист инспекции

- Источники:
  - все URL доступны, нет блокировок/капчи,
  - есть стабильные идентификаторы `source_id`.
- Дифф:
  - изменения не “шумят” из‑за меню/баннеров,
  - изменения не дублируются между источниками.
- База знаний:
  - статьи имеют метаданные, источники и даты,
  - legal‑разделы имеют повышенный приоритет и отдельную хронологию,
  - нет дублей “одной и той же темы” в разных секциях.
- Change package:
  - есть `kb_targets`, `links`, `category`, `severity`,
  - формулировки годятся для Content‑агента без угадываний.
- Логи и стоимость:
  - есть `request_id` сквозной,
  - есть учёт токенов и стоимость,
  - дорогие запросы объяснимы (top N).
